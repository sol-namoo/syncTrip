# SyncTrip

> A collaborative travel planning workspace built with Next.js, Supabase, Google Maps, and a 3D ticket-sharing concept.

SyncTrip is an MVP-stage product for planning trips with other people in one shared workspace. Users can create trips, search places, drag stops into day columns, see changes reflected on a map, and collaborate in real time through Supabase Realtime.

The MVP is currently in its final stretch of development, and you can try the live app at `https://sync-trip-zeta.vercel.app/`. Feedback of any kind is welcome.

## Learning Goals

- This project is also a hands-on learning project for trying new technologies in a product-shaped MVP.
- The core MVP learning goals are:
  - concurrent collaboration with websockets / realtime synchronization
  - basic usage of Three.js for an interactive 3D ticket viewer
  - practical SQL database usage with Supabase

## Beyond the MVP

- After the MVP, additional learning goals remain:
  - mobile UI support and layout refactoring
  - a mobile app using React Native
  - replacing Supabase with an explicit API layer and ORM (`Node.js + Prisma`)
  - gradually practicing a serverless / cloud-native architecture

## Architecture

SyncTrip follows a feature-driven structure. App Router files stay thin and mostly resolve auth, routing, and initial server-side data. Feature folders own their UI, hooks, Supabase queries, mutations, and local helpers.

```tsx
// src/app/workspace/[id]/page.tsx
const snapshot = await getWorkspaceSnapshot(id);
const role = resolveWorkspaceRole(user?.id, snapshot.members);

return (
  <WorkspaceScreen
    snapshot={snapshot}
    tripId={id}
    actor={actor}
  />
);
```

### Core patterns

- Server components fetch the initial workspace snapshot.
- Zustand stores hydrate client-side from that snapshot.
- React Query wraps standard client fetch and mutation flows.
- Supabase Realtime handles presence and activity broadcasts.
- Postgres RPC is used for transactional operations such as trip creation and card reordering.

## Database Model

The repository includes Supabase SQL migrations for the main collaboration model.

| Table / Function | Role |
| --- | --- |
| `trips` | Top-level trip metadata |
| `trip_members` | Membership and role mapping (`owner`, `editor`) |
| `trip_days` | Per-day columns used by the workspace board |
| `trip_items` | Planned places stored in bucket or day lists |
| `trip_share_settings` | One persistent share code and message per trip |
| `create_trip_with_owner` | RPC for creating a trip and owner membership atomically |
| `move_trip_item` | RPC for saving drag-and-drop ordering atomically |

<details>
<summary><strong>Schema notes</strong></summary>

- Row Level Security is enabled on the main collaboration tables.
- Realtime publication is used for workspace-related tables.
- The board model evolved from `day_index` to `trip_day_id`, and the codebase already reflects that newer direction.

</details>

## Main Features

### 1. Collaborative trip workspace

The workspace merges two views:

- A Kanban-like board for bucket and day planning
- A Google Map that visualizes selected places and route segments

### 2. Realtime presence, not raw cursor sync

SyncTrip deliberately broadcasts domain targets instead of viewport coordinates. That means collaborators see who is online, who is dragging, and who is editing, without relying on fragile pixel-perfect pointer sharing.

```ts
type WorkspaceBroadcastEvent =
  | { type: "target"; target: { kind: "card" | "column" | "none"; id?: string } }
  | { type: "drag"; state: "start" | "end"; itemId: string; columnId: string | null }
  | { type: "editing"; state: "start" | "end"; cardId: string };
```

### 3. Place search and route visualization

Google Places autocomplete is wrapped behind a feature adapter, then converted into normalized place details that can be inserted directly into the workspace.

```ts
const predictions = await autocompleteService.getPlacePredictions({
  input: normalized,
  types: ["establishment"],
});
```

### 4. 3D ticket sharing concept

The share modal turns a trip snapshot into a passport-style render model. It derives destination labels, route stops, participant counts, and a persistent share code for export.

### 5. Demo workspace

The `/workspace/demo` route signs visitors in anonymously and mounts a seeded workspace snapshot so the collaboration UI can be explored without creating a real trip first.

## Environment Variables

Create a local `.env.development` file with the following values:

```bash
NEXT_PUBLIC_SUPABASE_URL=your_supabase_project_url
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_DEFAULT_KEY=your_supabase_publishable_key
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=your_google_maps_api_key
NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID=your_google_maps_map_id
```

> [!WARNING]
> Without `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`, the workspace still loads but the map canvas falls back to an empty-state message.

## Local Setup Checklist

1. Install dependencies with `npm install`.
2. Add the required environment variables.
3. Apply Supabase migrations to your project.
4. Configure Supabase auth providers you want to use.
5. Enable Google Maps and Places for your API key.
6. Start the app with `npm run dev`.

## Example Usage

### Create a trip

1. Sign in at `/login`.
2. Open `/trips`.
3. Click the trip creation dialog.
4. Enter a title, dates, and one or more destinations.
5. You will be redirected into the new workspace.

### Plan the itinerary

1. Search for a place from the workspace search panel.
2. Insert it into the bucket or a day column.
3. Drag cards between days.
4. Watch markers and route segments update on the map.

### Share the ticket concept

1. Open the share action in the workspace header.
2. Add an optional message for the ticket back side.
3. Persist the share code.
4. Copy the generated public link preview.

## Contributing

There is no dedicated `CONTRIBUTING.md` file yet, so contributors should use the current project conventions:

- Keep routing concerns inside `src/app`.
- Put feature logic in `src/features/<feature>`.
- Prefer shared primitives in `src/components/ui`.
- Keep Supabase-facing code in feature `lib` folders or shared `src/lib`.
- Preserve the feature-driven structure instead of introducing cross-cutting folders too early.
- Run `npm run lint` before opening a pull request.
