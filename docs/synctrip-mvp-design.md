# SyncTrip MVP Screen Design Spec

## 섹션 1: Product UX Summary

### 핵심 사용자 시나리오
1. 사용자는 랜딩에서 서비스가 `실시간 협업 여행 플래너`와 `3D 여권 결과물` 중심이라는 점을 바로 이해하고, 로그인 없이 데모 성격의 화면을 둘러본 뒤 필요 시 로그인한다.
2. 로그인한 사용자는 `My Trips`에서 기존 여행 프로젝트를 확인하고, 제목/기간만 입력해 새 여행을 만든 뒤 즉시 Workspace로 진입한다.
3. 여러 사용자는 Workspace에서 `장소 검색 -> 바구니 저장 -> Day 컬럼으로 드래그` 흐름으로 일정을 같이 편집하고, 누가 접속 중인지와 누가 어떤 카드를 만지는지 실시간으로 확인한다.
4. 여행 일정이 정리되면 사용자는 `3D 여권 발급받기`를 눌러 결과 화면을 열고, 장소 스탬프가 찍힌 3D 여권을 회전해 본 뒤 이미지로 다운로드하거나 링크를 공유한다.

### 제품 UX 원칙
- MVP에서 사용자의 주 작업은 `일정 만들기`와 `결과 공유` 두 가지뿐이다.
- Workspace는 `조작성`이 우선이다. 예쁜 장식보다 `검색, 드래그, 충돌 인지, 현재 상태 확인`이 먼저 보여야 한다.
- 실시간 협업은 기능이 아니라 화면의 기본 상태처럼 보여야 한다.
- 3D 여권은 결과물의 하이라이트이므로 시각적으로 강하게, 하지만 제어는 단순하게 유지한다.
- 모바일은 `편집`보다 `확인/공유/다운로드`에 집중한다.

## 섹션 2: Information Architecture

### 전체 IA
- `/`
  - 랜딩
  - 로그인 CTA
  - 데모/읽기 전용 둘러보기 진입
- `/login`
  - 소셜 로그인
  - 로그인 후 이전 의도 경로로 복귀
- `/trips`
  - My Trips 대시보드
  - 새 여행 만들기 모달
- `/workspace/[tripId]`
  - 협업 일정 편집 메인 화면
  - 초대
  - 3D 여권 발급
- `/share/[tripId]`
  - 읽기 전용 공유 화면
  - 3D 여권 결과 중심
  - 모바일 최적화

### 화면 간 이동 규칙
- 비로그인 상태에서 `새 여행 만들기`, `참여`, `저장`, `초대`, `3D 여권 발급` 시 로그인 모달 호출
- 여행 생성 완료 시 `/workspace/[tripId]`로 즉시 이동
- Workspace 상단 `3D 여권 발급받기` 클릭 시 전체 페이지 이동보다 `Export Modal` 우선
- 공유 링크는 `/share/[tripId]`로 연결되고, 편집 권한이 없는 사용자는 읽기 전용만 가능

## 섹션 3: Screen Specs

### 1. 랜딩 & 로그인 페이지

#### 화면 목적
- 서비스의 핵심 가치 2개를 빠르게 전달한다.
- 로그인 전에도 “무슨 서비스인지”와 “어떤 결과가 나오는지”를 이해시키고 진입 장벽을 낮춘다.

#### 주요 사용자 행동
- 실시간 협업/3D 여권 특징 확인
- Google 또는 GitHub 로그인
- 로그인 없이 데모 둘러보기
- 일정 생성/참여 시 로그인 요구 확인

#### 레이아웃 구조
- 상단 고정 헤더
  - 좌측 로고 `SyncTrip`
  - 우측 `둘러보기`, `로그인`
- 히어로 섹션
  - 좌측 45%: 한 줄 가치 제안, 짧은 설명, CTA 2개
  - 우측 55%: `Workspace + 3D Passport`가 같이 보이는 합성 프리뷰
- 하단 보조 섹션 2개
  - 섹션 1: 실시간 협업 설명
  - 섹션 2: 3D 여권 결과물 설명
- 로그인 모달
  - 중앙 420px 폭
  - 상단 제목
  - 소셜 로그인 버튼 2개
  - 하단 설명: “로그인 후 여행 생성/참여 가능”

#### 핵심 컴포넌트
- 브랜드 헤더
- 히어로 카피 블록
- Primary CTA: `Google로 계속하기`
- Secondary CTA: `GitHub로 계속하기`
- Ghost CTA: `먼저 둘러보기`
- 데모 프리뷰 패널
  - 좌: 미니 칸반
  - 우: 미니 3D 여권
- 로그인 요구 모달

#### 상태 정의
- 기본: 랜딩 + CTA 노출
- 비로그인 둘러보기: 일부 화면 읽기 전용 접근 가능
- 로그인 모달 오픈: 배경 dimmed, 현재 액션 문맥 표시
- 로그인 처리 중: 소셜 버튼 disabled + spinner
- 로그인 실패: 모달 하단 인라인 에러
- 초대 링크 접근 시 비로그인: “참여하려면 로그인 필요” 메시지와 함께 모달 자동 오픈

#### 인터랙션/모션
- 히어로 우측 프리뷰는 자동 재생 애니메이션 1회만 실행
  - 카드가 Day 컬럼으로 이동
  - 3D 여권이 가볍게 회전
- 로그인 모달은 scale + fade 180ms
- CTA hover는 색 변화보다 elevation 변화 위주

#### 모바일 대응 여부
- 대응 필요
- 모바일에서는 히어로를 세로 적층
- 프리뷰는 정적 이미지 + 짧은 루프 애니메이션으로 축소
- 로그인 버튼은 풀폭

#### UI 판단 근거
- 랜딩의 목표는 정보 탐색이 아니라 `이 서비스가 무엇인지 5초 안에 이해`시키는 것이다.
- 기능 나열보다 `같이 편집한다`와 `결과가 여권으로 나온다` 두 축만 보여주는 것이 MVP 메시지에 맞다.

### 2. 내 일정 대시보드 (My Trips)

#### 화면 목적
- 사용자가 자신의 여행 프로젝트를 빠르게 찾고 생성하게 한다.

#### 주요 사용자 행동
- 여행 목록 탐색
- 새 여행 만들기
- 최근 편집한 여행 재진입
- 초대받은 여행 확인

#### 레이아웃 구조
- 상단 헤더
  - 좌측: `My Trips` 제목 + 보조 설명
  - 우측: Primary 버튼 `새 여행 만들기`
- 필터 영역
  - 탭 2개: `내가 만든 여행`, `초대받은 여행`
  - 정렬 드롭다운: `최근 수정순`
- 카드 그리드
  - 데스크톱 3열
  - 카드 높이 고정
- 생성 모달
  - 제목 입력
  - 시작일 / 종료일 2열
  - 하단 `취소`, `생성하고 편집 시작`

#### 핵심 컴포넌트
- 페이지 헤더
- Trip card
  - 여행 제목
  - 기간
  - 멤버 수
  - 마지막 수정 시간
  - 상태 배지 `내가 만든 여행` / `초대됨`
- Empty state 카드
- Create Trip modal
- Date range inputs

#### 상태 정의
- 목록 있음
- 완전 비어 있음
  - “첫 여행을 만들어보세요”
- 로딩 중
  - 카드 스켈레톤 6개
- 생성 모달 유효성 실패
  - 제목 없음
  - 종료일이 시작일보다 빠름
- 생성 처리 중
  - 제출 버튼 loading
- 초대받은 여행이 있으나 아직 열람 전
  - `새 초대` 점 배지

#### 인터랙션/모션
- 새 여행 카드 hover 시 살짝 상승
- 생성 성공 시 생성된 카드가 목록 최상단에 삽입되고 바로 Workspace로 전환
- 탭 전환은 underline slide 160ms

#### 모바일 대응 여부
- 대응 필요
- 1열 카드 리스트로 전환
- 생성 모달은 바텀시트 형태 권장

#### UI 판단 근거
- 대시보드는 관리 화면이지 편집 화면이 아니다. 카드 정보는 `재진입 판단`에 필요한 최소 정보만 노출한다.
- 생성 입력값은 PRD 범위대로 `제목/기간`만 유지해 진입 마찰을 줄인다.

### 3. 핵심 워크스페이스

#### 화면 목적
- 검색한 장소를 바구니에 담고, Day 단위 칸반으로 배치하면서 여러 사용자가 동시에 일정을 편집하게 한다.

#### 주요 사용자 행동
- 장소 검색
- 검색 결과를 바구니로 추가
- 바구니 카드를 Day 컬럼으로 이동
- Day 간 순서 변경
- 카드 메모 편집
- 동료 초대
- 3D 여권 발급

#### 레이아웃 구조
- 전체는 `상단 헤더 + 본문 좌우 스플릿`
- 상단 헤더 높이 72px
- 본문 높이 `calc(100vh - 72px)`

##### 상단 헤더
- 좌측
  - 여행 제목
  - 여행 기간
  - 저장 상태 텍스트 `모든 변경사항 저장됨` / `동기화 중`
- 중앙
  - 접속 중 아바타 스택
  - `+3` 형태의 추가 인원 집계
- 우측
  - Secondary `친구 초대`
  - Primary `3D 여권 발급받기`

##### 본문 좌우 비율
- 좌측 지도 45%
- 우측 보드 55%
- 사용자가 divider를 드래그해 40:60 ~ 50:50 범위에서 조절 가능
- 최소 폭
  - 지도 420px
  - 보드 560px

##### 좌측 지도 패널
- 상단 미니 툴바
  - `전체 보기`
  - 일자 필터 chips: `All`, `Day 1`, `Day 2`...
- 메인 지도
  - 선택된 카드/장소 마커 강조
  - 일자별 색 경로 표시
- 하단 요약 바
  - 선택 일자의 장소 수
  - 대략 이동 흐름 시각 요약

##### 우측 보드 패널
- 수평 스크롤 구조
- 컬럼 구성
  - 첫 번째 고정 컬럼: `장소 바구니`
  - 이후 스크롤 컬럼: `Day 1`, `Day 2`, `Day 3`...
- 각 컬럼 폭 320px
- 컬럼 간 간격 16px
- 보드 상단 보조 바
  - 검색창
  - 검색 결과 개수
  - `Day 추가` 버튼은 MVP 범위에서 필요 시만 노출, 기본은 생성 시 기간 기준 자동 생성
- 컬럼 헤더
  - 컬럼명
  - 카드 수
  - 일자 색상 인디케이터
- 컬럼 본문
  - 카드 리스트
  - 빈 컬럼일 경우 drop hint
- 컬럼 하단
  - 해당 일자 총 장소 수
  - 간단한 note 없음, 메모는 카드 단위만 허용

##### 카드 구조
- 상단 행
  - 좌측 썸네일 56x56
  - 우측 텍스트 블록: 장소명 2줄, 주소 2줄
- 중단
  - 메모 입력 textarea 2줄 기본 높이
- 하단 행
  - 좌측 상태 칩
    - `검색 결과 추가`
    - `편집 중`
    - `동기화됨`
  - 우측 아이콘 버튼 `삭제`
- 카드 전체가 드래그 핸들 역할을 가지되, 메모 입력 영역은 텍스트 편집 우선

#### 핵심 컴포넌트
- Workspace header
- Presence avatar stack
- Invite button
- Export CTA
- Split pane divider
- Map toolbar
- Day filter chips
- Map marker
- Route polyline
- Search input
- Search result list dropdown
- Basket column
- Day column
- Sortable place card
- Memo textarea
- Conflict badge
- Multi-cursor overlay
- Active card editor indicator
- Save status label
- Invite modal

#### 상태 정의

##### 기본 편집 상태
- 장소 없음
  - 바구니 empty illustration
  - 검색 유도 문구
- 바구니에만 카드 있음
  - 지도에는 단일 마커만 표시
- Day 컬럼 배치 완료
  - 일자별 색 경로 표시

##### 검색 상태
- 입력 전
  - placeholder: `도시나 장소를 검색하세요`
- 입력 중
  - debounce 300ms
- 결과 로딩
  - dropdown skeleton 5개
- 결과 없음
  - “검색 결과가 없습니다”
- 추가 완료
  - 선택 결과가 바구니 상단에 삽입되고 토스트 1.5초

##### 실시간 협업 상태
- 누가 접속 중인지
  - 헤더 아바타 스택에 실시간 표시
  - hover 시 `이름 + 현재 보고 있는 Day`
- 누가 어떤 카드를 만지는지
  - 카드 우상단에 해당 사용자 색상 점 + 이름 pill
  - 메모 입력 중이면 카드 테두리도 같은 색으로 강조
- 충돌 가능성이 있을 때
  - 내가 열어둔 카드에 다른 사용자가 편집 시작하면 상단 인라인 경고 배너
  - 문구: `지금 Alex가 이 카드를 수정 중입니다. 저장 시 마지막 수정이 반영됩니다.`
  - 메모 입력창 우측 상단에 `공동 편집 중` 배지
- 내 행동이 다른 사람에게 어떻게 반영되는지
  - 카드 이동 즉시 상대 화면에서 같은 애니메이션으로 이동
  - 내가 집는 순간 카드가 상대 화면에서 반투명 + `Minji 이동 중`
  - 메모 입력은 blur 기준 저장이 아니라 500ms throttled sync
  - 다른 사람의 수정 내용은 카드 하단에 짧게 `Alex가 메모 업데이트` 표시 후 자연 소멸

##### 저장/동기화 상태
- 로컬 변경 있음
  - 헤더에 `동기화 중...`
- 서버 반영 완료
  - `모든 변경사항 저장됨`
- Realtime 연결 끊김
  - 상단 노란 배너 `실시간 연결이 끊겼습니다. 변경사항은 재연결 후 동기화됩니다.`
- 저장 실패
  - 빨간 점 + `저장 실패, 다시 시도`

##### 권한 상태
- 편집 가능
- 읽기 전용 참여자
  - 카드 드래그 비활성
  - 메모 readonly
  - `보기 전용` 배지 노출

#### 인터랙션/모션
- 드래그 시작
  - 카드 scale 1.02
  - 그림자 상승
  - 원래 위치에는 dashed placeholder
- 드래그 중
  - 드롭 가능 컬럼 배경 tint
  - 지도에서 해당 장소 마커 pulse
- 드롭 완료
  - 카드가 새 위치에 snap-in
  - 지도 경로가 250ms 안에 다시 그려짐
- 메모 편집 중
  - 카드 높이 자동 확장
  - 저장 중 미세한 점 세 개 애니메이션
- 다중 커서
  - 이름표 포함 작은 컬러 커서
  - 지도/보드 영역 모두에서 보이되 2초 무동작 시 opacity 감소
- 초대 버튼 클릭
  - 작은 모달에서 링크 복사
- `3D 여권 발급받기`
  - 클릭 시 배경을 유지한 채 full-screen modal 오픈
  - 현재 보드 데이터 스냅샷 기준 렌더링 시작

#### 모바일 대응 여부
- 완전한 편집 화면으로는 비권장
- 모바일에서는 읽기 전용 또는 제한 편집
  - 지도/보드 탭 전환형
  - 드래그 앤 드롭 대신 `일자에 추가` 액션시트 사용 가능
- MVP 우선순위상 데스크톱 메인, 모바일은 확인용 보조 경험

#### UI 판단 근거
- 지도와 보드를 동시에 보여주는 이유는 `카드 배치 결과가 공간적으로 어떻게 보이는지` 즉시 확인시키기 위해서다.
- 장소 바구니를 Day 컬럼과 같은 보드 안에 넣는 이유는 사용자가 검색 결과와 일정 편집을 한 작업면에서 끝내게 하기 위해서다.
- 실시간 협업 정보는 별도 패널이 아니라 `헤더, 카드, 커서`에 분산 배치해야 작업 흐름을 끊지 않는다.
- 충돌 제어는 락 기반보다 `현재 누가 수정 중인지 강하게 노출`하는 방식이 MVP 구현과 사용성 모두에 유리하다.
- 저장 여부를 토스트로만 처리하면 놓치기 쉽다. 상단 헤더의 지속 상태 텍스트가 더 적합하다.

### 4. 3D 여권 결과 화면 (Export Modal)

#### 화면 목적
- 완성된 여행을 `결과물`로 경험하게 하고, 가장 먼저 다운로드하게 만든다.

#### 주요 사용자 행동
- 3D 여권 회전
- 여행 결과 미리보기 확인
- 이미지 다운로드
- 공유 링크 사용

#### 레이아웃 구조
- 데스크톱은 전체 화면 모달
- 중앙 정렬 3단 구조
  - 좌측 20%: 여행 요약
  - 중앙 60%: 3D 오브젝트 뷰어
  - 우측 20%: 액션 패널
- 모바일은 세로 1열
  - 상단 3D 뷰어
  - 하단 액션 시트

##### 중앙 3D 뷰어
- 큰 여권 커버 + 내부 페이지 + 보딩패스 느낌의 카드 조합
- 기본 카메라는 3/4 시점
- 사용자는 드래그로 회전 가능
- 더블클릭 시 정면 보기 복귀

##### 좌측 여행 요약
- 여행 제목
- 기간
- 참여자 수
- 방문 도시 수
- 대표 장소 3개 텍스트 목록

##### 우측 액션 패널
- 최상단 Primary CTA: `이미지 다운로드`
- Secondary CTA: `링크 공유`
- Tertiary CTA: `다시 편집`
- 하단 안내: “현재 보이는 각도로 이미지가 저장됩니다”

#### 핵심 컴포넌트
- Full-screen export modal
- 3D passport viewer
- Stamp cluster
- Orbit controls hint
- Download button
- Share link button
- Copy success toast
- Render loading overlay
- Read-only share header

#### 상태 정의

##### 렌더링 상태
- 초기 로딩
  - 중앙 spinner 대신 여권 실루엣 skeleton
  - 문구: `여권을 발급하는 중...`
- 렌더링 완료
  - 회전 가능
- 렌더 실패
  - 정적 포스터 이미지 fallback + 다시 시도 버튼

##### 다운로드 상태
- 기본
- 생성 중
  - 버튼 loading
- 완료
  - 짧은 success toast
- 실패
  - 인라인 에러

##### 공유 상태
- 링크 복사 전
- 복사 완료
  - 버튼 텍스트 `링크 복사됨`
- 비로그인 공개 접근
  - 읽기 전용 공유 화면으로 동일 구조 제공

#### 인터랙션/모션
- 모달 오픈 시 배경 blur + 여권이 전면으로 슬라이드 인
- 3D 오브젝트는 idle 상태에서 아주 느린 부유 motion
- 사용자가 회전 중일 때 idle motion 정지
- 다운로드 성공 시 플래시형 효과 없이 버튼 상태만 명확히 변경

#### 모바일 대응 여부
- 반드시 대응
- 모바일은 3D 뷰어 높이 45vh 확보
- CTA는 하단 고정 액션 바
- 다운로드 버튼을 가장 크게 유지

#### 3D 여권 화면 필수 설계 결정
- 렌더링 오브젝트의 시각적 컨셉
  - `진짜 여권 표지 + 내부 스탬프 페이지 + 보딩패스 한 장`의 혼합 오브젝트
  - 너무 사실적인 모델보다, 형태는 명확하고 스탬프가 잘 읽히는 반-스타일라이즈드 방향
- 스탬프 배치 규칙
  - Day 순서 기준으로 좌상단에서 우하단으로 흐르게 배치
  - 같은 Day의 장소는 같은 색군 사용
  - 도시 단위 대표 스탬프 1개 + 세부 장소는 소형 보조 스탬프로 처리
  - 8개 이상 장소는 전부 텍스트로 넣지 않고, 대표 장소 우선 배치
- 다운로드 CTA 우선순위
  - 가장 크고 가장 먼저 보이는 버튼
  - 우측 패널 최상단, 진한 채움 스타일
- 공유 버튼의 시각적 위계
  - 다운로드보다 한 단계 낮은 secondary 버튼
  - 링크 공유는 보조 행동으로 배치
  - 소셜 아이콘 나열보다 `링크 복사` 1개가 MVP에 더 적합

#### UI 판단 근거
- 결과 화면의 핵심은 감상보다 `저장`이다. CTA 위계는 다운로드가 명확히 1순위여야 한다.
- 복잡한 공유 옵션을 늘리기보다, 링크 복사 1개로 끝내는 편이 MVP 범위와 구현 비용에 맞다.
- 스탬프는 정보 밀도보다 `여행을 다녀온 기록처럼 보이는 인상`이 중요하다.

## 섹션 5: States and Edge Cases

### 인증 관련
- 비로그인 사용자가 생성/참여/편집 시도
  - 현재 맥락 유지한 채 로그인 모달 오픈
- 로그인 후 원래 보던 trip 또는 초대 링크로 복귀

### 여행 생성 관련
- 시작일/종료일 미입력
- 종료일이 시작일보다 빠름
- 제목이 너무 짧음
- 생성 성공 후 Workspace 이동 실패
  - 생성은 되었지만 이동 실패 시 `여행으로 이동` 버튼 제공

### 검색/카드 관련
- 장소 검색 API 응답 지연
- 같은 장소를 여러 번 추가
  - 완전 차단보다 `이미 추가됨` 배지와 재추가 허용 여부 명시
  - MVP에서는 중복 허용 가능, 대신 카드 구분 필요
- 썸네일 없는 장소
  - 기본 placeholder 이미지
- 주소가 너무 긴 경우
  - 2줄 ellipsis

### 협업 관련
- 두 사용자가 동시에 같은 카드 메모 수정
  - 마지막 저장 우선
  - 직전 변경자 표시
  - 내 로컬 입력이 덮일 수 있음을 사전 경고
- 한 사용자가 드래그 중 연결 끊김
  - 카드 원위치 복귀 또는 마지막 서버 상태로 재동기화
- Presence 유저가 갑자기 사라짐
  - 아바타와 커서 3초 내 페이드아웃

### Export 관련
- 장소가 하나도 없는 여행
  - 여권 발급 CTA 비활성
  - 툴팁: `최소 1개 장소가 필요합니다`
- 렌더링은 됐지만 다운로드 캡처 실패
  - 정적 fallback 이미지 다운로드 제공
- 모바일 저사양 디바이스
  - 3D 품질 한 단계 낮춤
  - 그림자/반사 감소

## 섹션 6: Motion and Realtime Feedback

### 모션 원칙
- 화려한 전환보다 `상태 변화 인지`가 목적이어야 한다.
- 드래그, 저장, 동기화, 충돌은 모두 별도 텍스트 없이도 움직임으로 감지 가능해야 한다.
- 모션 시간 기준
  - hover 120ms
  - modal 180ms
  - drag settle 200ms
  - realtime 반영 150~250ms

### 실시간 협업 피드백 규칙
- 누가 접속 중인지
  - 헤더 아바타 스택으로 항상 노출
  - 6명 초과 시 `+N`
- 누가 어떤 카드를 만지는지
  - 카드 테두리 색상 + 이름 pill
  - 해당 사용자의 커서 색과 동일하게 통일
- 충돌 가능성이 있을 때 어떻게 보여줄지
  - 카드 상단 warning stripe
  - 메모 필드 우측 상단 `공동 편집 중`
  - 상단 배너로 마지막 저장 우선 정책 안내
- 내 행동이 다른 사람에게 어떻게 반영되는지
  - 카드 이동, 메모 수정, 삭제가 상대 화면에 바로 반영
  - 이동은 텔레포트가 아니라 짧은 위치 전환 애니메이션으로 보여줘야 함
  - 삭제는 바로 사라지지 말고 120ms 축소 후 제거

### 구체 모션 제안
- 지도 마커
  - 선택 시 1회 pulse
  - 동일 Day 마커는 색군 통일
- 카드
  - drag overlay는 원본보다 그림자 강하게
  - drop placeholder는 점선 박스
- Export 여권
  - 모달 오픈 시 z축 앞으로 나오며 살짝 회전
  - idle float는 4~6초 주기, 미세하게만 적용

## 섹션 7: Designer-to-Developer Handoff Notes

- 색 체계는 기능 중심으로 고정한다.
  - `Day 색상`은 데이터 시각화 용도
  - `Presence 색상`은 사용자 식별 용도
  - 둘을 섞지 않는다
- Workspace에서 가장 중요한 정보 우선순위는 다음 순서다.
  - 현재 여행 제목과 저장 상태
  - 접속 중 사용자
  - 장소 검색과 카드 배치 상태
  - 지도 결과
- 보드 컬럼 수는 여행 기간에 맞춰 자동 생성한다.
  - 예: 3일 여행이면 `Day 1~3`
  - MVP에서는 사용자 임의 추가/삭제는 후순위로 둬도 된다
- 메모 저장 방식은 입력마다 전체 즉시 저장보다 `짧은 throttle + blur 시 강제 flush`가 안전하다.
- 충돌 해결은 CRDT 수준 설계보다 `누가 편집 중인지 강하게 노출`하는 것으로 MVP를 마감하는 편이 적절하다.
- 검색 결과 패널은 별도 페이지로 보내지 말고 Workspace 내부 드롭다운에서 끝내야 한다.
- 3D 여권 렌더링 데이터는 Workspace 실시간 상태 전체가 아니라 `발급 시점 스냅샷`을 사용한다.
- 다운로드는 현재 카메라 각도 기반 이미지 저장이 구현 비용이 가장 낮다.
- 모바일에서 편집 UX를 무리하게 맞추지 않는다.
  - MVP 기준 모바일은 `공유 결과 확인`과 `다운로드`가 주 역할이다.
- 접근성 최소 기준
  - 드래그 불가능 환경을 위해 카드에 `일자로 이동` 메뉴 대체 제공 고려 가능
  - 버튼/배지/경고 상태는 색만으로 구분하지 않는다
